# PWS Cup 事例アーカイブ — 運用ガイド

このディレクトリは https://pwscup.github.io/pwssite/about/ で公開されている「PWS Cup 事例アーカイブ」ページのソースです。

## ファイル構成

| ファイル | 役割 |
|---|---|
| `index.html` | ページ本体。`cases.json` を fetch して描画する |
| `cases.json` | **事例データ（編集対象はここだけ）** |
| `cases.schema.json` | JSON Schema。バリデーションに使用 |
| `README.md` | 本ファイル |

## 事例を追加・編集する手順

### 通常フロー（GitHub Web UI で完結）

1. **GitHub で `about/cases.json` を開く** → 鉛筆アイコンで編集モードに入る
2. **`cases` 配列の末尾に1要素追加** — フォーマットは下の「データスキーマ」を参照
3. **コミット**（直接 gh-pages にコミット、または PR を作成）
4. **GitHub Actions が自動でバリデーション実行**
   - ✅ 成功 → そのままデプロイ → 数分で公開ページに反映
   - ❌ 失敗 → デプロイが止まる、コミット者にメール通知、コミット横に赤 ✗ が表示される
5. 失敗した場合はエラーメッセージに従って修正、再度コミット

### LLM を使った楽な追加方法

JSONを手で書くのが面倒な場合、ChatGPT/Claude/Gemini 等に **下のプロンプト** をコピペして使ってください。LLM が必要事項を順に質問してくれて、最後に貼り付け可能なJSONブロックを出力します。

> ```
> あなたは PWS Cup 事例アーカイブのデータ入力アシスタントです。
> 私が紹介したい事例について、以下のスキーマに従って JSON 1件分を出力してください。
> 不足情報は1つずつ私に質問し、すべて揃ったら最後にJSONブロックを単独で出力してください。
>
> ## スキーマ（1件分）
> ```
> {
>   "no": <整数：連番。「現在の最大値+1」を私に聞くか、わからなければ仮で 999 を入れる>,
>   "year": <整数：開催年（西暦）2015以降>,
>   "category": "<文字列：細分類。例『公式ページ』『企業公式・受賞報告』『研究室ニュース』『Qiita・参加記』など、過去の傾向に合わせる>",
>   "title": "<文字列：記事/ページのタイトル（記事の見出しそのまま）>",
>   "source": "<文字列：掲載元。例『PWS公式』『NSSOL』『明治大学 菊池研究室』『Qiita / NTTドコモ R&D』>",
>   "type": "<official|company|university|paper|blog のいずれか>",
>   "award": "<文字列：受賞名（任意）。例『総合優勝』『ベストアタック賞』『攻撃3位』。なければ省略>",
>   "url": "<文字列：記事URL（任意、http(s)で始まる）。なければ省略>"
> }
> ```
>
> ## type の決め方
> - `official`: PWS 公式サイト・公式資料・PWS Meetup 資料など
> - `company`: 企業の公式ニュース・PR・技術ブログ・受賞報告
> - `university`: 大学・研究室の公式ニュース・受賞報告
> - `paper`: 論文・学会発表・研究発表資料
> - `blog`: 個人ブログ・Qiita・note・SlideShare など個人/コミュニティ寄りの記事
>
> ## ルール
> - 任意項目（award/url）は値がないときは出力に含めない（null や空文字は不可）
> - JSON はコードブロックで囲んで出力
> - 確認のため、最後に簡単な要約（年度／type／タイトル）を日本語で1行添える
>
> では、紹介したい事例について教えてください。まずは何の事例ですか？
> ```

LLMの出力JSONを `cases.json` の `cases` 配列の末尾にカンマ区切りで貼り付けてください。

### `no` の連番について

- 通常は **`cases` 配列の最後の要素の `no` + 1** を使います
- 過去に使った `no` を再利用するとバリデーションが失敗します（`no` は重複不可）
- `no: 13` のように歯抜けの番号があっても問題ありません（連続性は不要）

## データスキーマ（リファレンス）

各事例の最小例：
```json
{
  "no": 54,
  "year": 2026,
  "category": "公式ページ",
  "title": "PWS Cup 2026 開催案内",
  "source": "PWS公式",
  "type": "official"
}
```

受賞 + URL付きの例：
```json
{
  "no": 55,
  "year": 2026,
  "category": "企業公式・受賞報告",
  "title": "○○がPWS Cup 2026で総合優勝",
  "source": "○○株式会社",
  "type": "company",
  "award": "総合優勝",
  "url": "https://example.com/news/pwscup2026"
}
```

正式な定義は `cases.schema.json` を参照。

## 年度テーマ（themes）の追加・更新

各年度のコンテストで何がテーマだったかを `themes` 配列に登録すると、年度ヘッダの下に表示されます。

### スキーマ
```json
{
  "year": 2026,
  "title": "テーマの一行サマリ",
  "description": "1〜2文の補足説明（任意）",
  "url": "https://www.iwsec.org/pws/2026/cup26.html",
  "sample": false
}
```

- `year`: 開催年（必須、重複不可）
- `title`: 一行テーマ（必須）
- `description`: 補足（任意、省略可）
- `url`: 公式ページ（任意、省略可）
- `sample`: `true` にすると公開ページに **「サンプル」バッジ** が表示される。**裏取りせずに仮置きしたデータには付ける**こと。確認後に外す（または削除）

### 既存サンプルの更新フロー
現在 2015〜2025 のテーマには `"sample": true` が付いています。各年度の運営担当が裏取り後、内容を確認・修正し `sample` フィールドを削除してください。

### LLM 用プロンプト（テーマ作成補助）
> ```
> あなたは PWS Cup 各年度のテーマ情報を整理するアシスタントです。
> 私が指定した年度について、以下のスキーマで JSON 1件分を出力してください。
> 不明な点は私に質問し、自信のある情報のみを採用してください。
>
> ## スキーマ
> ```
> {
>   "year": <整数：開催年>,
>   "title": "<文字列：その年のコンテストの一行テーマ。題材・指標・特徴のいずれかを核に20文字前後で>",
>   "description": "<文字列：1〜2文の補足。題材データ・主な評価指標・新設賞などを簡潔に>",
>   "url": "<文字列：その年のPWS Cup公式ページURL（推測しないこと、確実なものだけ）>",
>   "sample": <真偽値：裏取り未完なら true、確認済みなら省略 or false>
> }
> ```
>
> 私が「2026年のテーマ」と言ったら聞き取り → JSON出力、を繰り返します。
> ```

## ローカルで検証する

```bash
# プロジェクトルートで実行
uv run python3 scripts/validate_cases.py
```

エラーがあれば全件まとめて表示されます。

## バリデーションの仕組み

GitHub Actions の `validate-cases` ジョブが以下を行います（`.github/workflows/convert_markdown.yml`）：

1. `about/cases.json` を JSON Schema (`cases.schema.json`) で検証
2. `cases` 配列内の `no` 重複を検出
3. `themes` 配列内の `year` 重複を検出
4. **どれか失敗するとデプロイがブロックされ、コミット者に GitHub からメール通知**

通るまでは公開ページのデータは更新されません（前回成功時のままです）。
